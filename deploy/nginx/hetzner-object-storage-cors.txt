# Hetzner Object Storage — bucket CORS and cache configuration
#
# Apply these settings in the Hetzner Cloud Console → Object Storage → Bucket Settings
# or via the S3 API (boto3 / aws-cli / MinIO client).
#
# ── CORS policy ──────────────────────────────────────────────────────────────
# Required so Flutter web players and browsers can fetch HLS segments directly.
# Apply to BOTH the upload bucket and the delivery bucket.
#
# AWS S3 CLI equivalent:
#   aws s3api put-bucket-cors \
#     --bucket ichebo-delivery \
#     --cors-configuration file://cors.json \
#     --endpoint-url https://fsn1.your-objectstorage.com

# cors.json:
{
  "CORSRules": [
    {
      "AllowedOrigins": ["*"],
      "AllowedMethods": ["GET", "HEAD"],
      "AllowedHeaders": ["*"],
      "ExposeHeaders": ["ETag", "Content-Length"],
      "MaxAgeSeconds": 3600
    }
  ]
}

# ── Cache-Control on object upload ───────────────────────────────────────────
# Hetzner Object Storage does not support CDN cache-rule overrides the way
# AWS CloudFront does. Cache headers must be set at upload time via the
# Content-Type and Cache-Control metadata on each PUT.
#
# The Go Video Engine (storage/s3.go PutObject) already accepts these via
# the s3.PutObjectInput. Add cache metadata per object type:
#
#   Live .m3u8 manifests:  Cache-Control: no-store, no-cache, must-revalidate
#   Live .ts segments:     Cache-Control: public, max-age=2
#   VOD .m3u8 manifests:   Cache-Control: public, max-age=3600
#   VOD .ts segments:      Cache-Control: public, max-age=2592000, immutable
#   Thumbnails (.jpg):     Cache-Control: public, max-age=86400
#
# Implementation: update s3.go PutObject to accept an optional cacheControl
# string parameter and set input.CacheControl when provided.
# The worker.go uploadProfileOutput call sets VOD cache headers.
# The stream archiver sets live segment headers.
