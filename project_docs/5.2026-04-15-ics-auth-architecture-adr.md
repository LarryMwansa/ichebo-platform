# ICS Platform — Authentication Architecture Decision
## Architectural Decision Record + Implementation Plan

> **Version:** 1.0 — 2026-04-15
> **Status:** LOCKED
> **Scope:** All API and web auth flows through MVP and production launch.
> DRF endpoints use Token auth. Django template views use session auth.
> Migration path to JWT documented for post-MVP.

---

## Part 1 — Architectural Decision

### 1.1 The Decision

The ICS Platform uses **Token Authentication** (Django REST Framework `authtoken`) for all DRF `/api/` endpoints. Django template views use **Session Authentication** (`django.contrib.sessions`). This two-tier approach serves the current HTMX MVP without requiring JWT implementation, while leaving a clear migration path for future mobile clients.

### 1.2 Auth Stack Summary

| Layer | Technology | Use Case | Auth Method |
|-------|-----------|----------|------------|
| **Web UI** | Django templates + HTMX | Browser sessions | `django.contrib.sessions` + `@login_required` |
| **DRF API** | Django REST Framework | Mobile, integrations | `rest_framework.authtoken` + DRF TokenAuthentication |
| **Future Mobile** | (Post-MVP) | iOS/Android clients | JWT (migration plan in Part 5) |

### 1.3 Token vs JWT Trade-off Analysis

#### Why Token Auth (Now)

| Criterion | Token Auth | JWT |
|-----------|-----------|-----|
| **Revocability** | ✅ Instant — delete token from DB, user loses access | ❌ Token valid until expiry; revocation requires blacklist |
| **Implementation** | ✅ Built into DRF; zero additional dependencies | ❌ Requires `djangorestframework-simplejwt` + configuration |
| **Stateless** | ❌ Requires DB lookup per request | ✅ Signature verification only; no DB hit |
| **Scope for MVP** | ✅ Only web + internal API (no mobile) | ❌ Overkill for session-based web UI |
| **Security (at rest)** | ✅ Server controls token storage (DB) | ❌ Client stores token (localStorage = XSS risk) |
| **Security (in transit)** | 🟡 HTTP `Authorization: Token {token}` | 🟡 HTTP `Authorization: Bearer {jwt}` |
| **Integration friction** | ✅ Third-party integrations expect Bearer tokens | 🟡 Some expect JWTs, but both are Bearer variants |

**Decision:** Token Auth for MVP. All web requests use session cookies (revoked on logout). All `/api/` requests use Token auth (stored in DB, deleted on logout or explicit revocation).

### 1.4 Authentication Flows — Current Implementation

#### Flow A: Web UI Registration + Login (Session Auth)

```
1. User visits /accounts/register/
2. User submits form → Django views.RegisterView
3. RegisterView creates User, calls django.contrib.auth.login(request, user)
4. Login sets SessionMiddleware cookie: Set-Cookie: sessionid={hash}
5. Browser includes sessionid cookie on all subsequent requests
6. @login_required checks request.user; HTMX views access user directly
7. User clicks logout → SessionMiddleware clears sessionid
```

**Security:** Session cookie is `HttpOnly`, `Secure` (production only), `SameSite=Lax`.

#### Flow B: DRF API Login (Token Auth)

```
1. Client POSTs /api/auth/login/ with email + password
2. DRF endpoint calls django.contrib.auth.authenticate()
3. Token model looks up or creates Token for user
4. Response: { "token": "9944b09199c62bcf9418ad846dd0e4bbdfc6ee4b" }
5. Client stores token in localStorage or in-memory (app memory preferred)
6. Client includes on all subsequent API calls: Authorization: Token {token}
7. DRF TokenAuthentication middleware validates token
8. Client calls /api/auth/logout/ → Token is deleted from DB
```

**Security:** Token is a 40-character SHA-1 hash; no payload; revocation is instant.

#### Flow C: Membership Request Approval (Cross-Flow)

When a steward approves a membership request, the system:
- Runs in **web view context** (steward is logged in via session)
- Creates a `UserPermission` via Django ORM (not via DRF)
- Steward's session is unchanged; new user gets a Token only when they log in via `/api/auth/login/`

**No cross-flow complexity:** Web views never call `/api/` endpoints. Web uses ORM; `/api/` uses DRF serializers.

### 1.5 Token Lifespan & Expiry

**Current implementation (MVP):**
- Token is created on login, no expiry time
- Token is deleted on logout or account suspension
- No rotation; same token valid for session lifetime

**Security consideration:**
- For MVP (internal team + early users): acceptable
- For production (wider user base): add token expiry + rotation (see Part 5, Post-MVP Improvements)

### 1.6 Token Storage & XSS Protection

**Web UI (Session):**
- Session cookie stored by browser (inaccessible to JS)
- HttpOnly flag prevents XSS from reading it
- CSRF token sent via form/HTMX header

**DRF API (Token):**
- Token sent via `Authorization: Bearer` header
- Never stored in `localStorage` (XSS vulnerable)
- If building mobile app: store in secure device storage (iOS Keychain, Android Keystore), **not** localStorage

### 1.7 Logout & Token Revocation

**Web logout:**
```python
# accounts/views.py — Django built-in
from django.contrib.auth.views import LogoutView
path('logout/', LogoutView.as_view(next_page='/accounts/login/'), name='logout'),
```
Session cookie is cleared; user is unauthenticated.

**DRF logout:**
```python
# api/auth/views.py
@login_required
def logout(request):
    """Delete user's token on logout."""
    Token.objects.filter(user=request.user).delete()
    return Response({"message": "Logged out."})
```
Token is deleted from DB; further API requests with that token are rejected.

**Account suspension:**
```python
# tenants/models.py or accounts/models.py
def suspend_user(user):
    """Revoke all tokens for a user."""
    user.is_active = False
    user.save()
    Token.objects.filter(user=user).delete()
```
User is marked inactive; all their tokens are deleted.

### 1.8 DRF Token Configuration

**In `settings/base.py`:**
```python
INSTALLED_APPS = [
    ...
    'rest_framework',
    'rest_framework.authtoken',
    ...
]

REST_FRAMEWORK = {
    'DEFAULT_AUTHENTICATION_CLASSES': [
        'rest_framework.authtoken.authentication.TokenAuthentication',
    ],
    'DEFAULT_PERMISSION_CLASSES': [
        'rest_framework.permissions.IsAuthenticated',
    ],
}
```

**In `api/auth/urls.py`:**
```python
from rest_framework.authtoken.views import obtain_auth_token

urlpatterns = [
    path('login/', obtain_auth_token, name='api-token-auth'),  # POST email+password → token
    path('logout/', views.logout, name='api-logout'),           # POST → delete token
]
```

**In `api/models.py` or signal handler:**
```python
from django.db.models.signals import post_save
from rest_framework.authtoken.models import Token

@receiver(post_save, sender=User)
def create_auth_token(sender, instance=None, created=False, **kwargs):
    """Create a token for every new user."""
    if created:
        Token.objects.create(user=instance)
```

**Note:** This is automatic in `djangorestframework==3.17.1` (already in `requirements.txt`). Tokens are created on User creation.

## Part 2 — Why Not JWT (For MVP)

### 2.1 JWT Deferral Rationale

| Reason | Impact |
|--------|--------|
| **Mobile not in MVP scope** | JWT is designed for stateless mobile clients. Web + internal API don't need it. |
| **Revocation is critical** | User suspension must be instant. JWT requires blacklist (defeats statelessness). Token auth uses DB delete (one query). |
| **No additional dependency needed** | Token auth is built into DRF. JWT requires `djangorestframework-simplejwt` (not in `requirements.txt`). |
| **HTMX doesn't call `/api/`** | Web views use ORM directly. Only `/api/` clients use auth tokens. Scope is small. |
| **All existing tests use Token auth** | 19 tests written for `accounts/tests.py` use `APITestCase` with Token auth. Migration would break them. |
| **Third-party integrations flexible** | Both Token and JWT are Bearer schemes. Swap is non-breaking for client code. |

### 2.2 Risks of Premature JWT

| Risk | Mitigation |
|------|-----------|
| Adds complexity before need | Don't add it. Migration is straightforward when mobile app starts. |
| Requires expiry handling | Token auth doesn't require rotation logic yet. Can add later. |
| Requires refresh token logic | Also deferred. Not needed for web sessions or single-user API clients. |
| Client-side token storage | HTMX doesn't do client-side storage. Mobile app will have secure storage. Not our concern yet. |

## Part 3 — Current Token Auth Implementation

### 3.1 Database Schema

```python
# rest_framework.authtoken.models.Token (already in DRF)
class Token(models.Model):
    key = models.CharField(max_length=40, primary_key=True)  # SHA-1 hash
    user = models.OneToOneField(User, on_delete=models.CASCADE, related_name='auth_token')
    created = models.DateTimeField(auto_now_add=True)
```

One token per user. Deleting the token revokes access.

### 3.2 DRF Endpoints (Existing)

| Endpoint | Method | Request | Response |
|----------|--------|---------|----------|
| `/api/auth/login/` | POST | `{ "username": "...", "password": "..." }` | `{ "token": "9944b09..." }` |
| `/api/auth/logout/` | POST | `Authorization: Token {token}` | `{ "message": "Logged out." }` |
| Any DRF view | GET/POST/etc | `Authorization: Token {token}` | Protected by `IsAuthenticated` permission |

### 3.3 Token Usage in Tests

All `APITestCase` tests use Token auth:

```python
# accounts/tests.py
class LoginAPITests(APITestCase):
    def test_login_returns_token(self):
        response = self.client.post('/api/auth/login/', {
            'username': self.user.email,
            'password': 'testpass123'
        })
        self.assertIn('token', response.json())
        self.token = response.json()['token']

    def test_authenticated_request_with_token(self):
        response = self.client.get(
            '/api/records/',
            HTTP_AUTHORIZATION=f'Token {self.token}'
        )
        self.assertEqual(response.status_code, 200)
```

## Part 4 — Deployment & Security (Production)

### 4.1 Token Secret Management

**In `settings/production.py`:**
```python
# Token keys are generated automatically; no additional secret needed.
# DRF's Token uses Django's SECRET_KEY for token signing (if custom salt is added).
# Default behavior: tokens are random 40-char hex strings — not cryptographically derived from SECRET_KEY.

# Standard practice: ensure SECRET_KEY is strong and unique per environment
SECRET_KEY = os.environ.get('SECRET_KEY')  # from .env
```

### 4.2 HTTPS & Token in Transit

**Requirement:** All `/api/` endpoints must use HTTPS in production.

```python
# settings/production.py
SECURE_SSL_REDIRECT = True
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True

# Token is sent in Authorization header (not in URL or cookie), so HTTPS is essential
```

### 4.3 Token Logging

**Do NOT log tokens.** Redact them from logs:

```python
# ics_project/logging_config.py or settings/production.py
LOGGING = {
    'filters': {
        'redact_tokens': {
            '()': 'logging_config.RedactTokensFilter',
        },
    },
    'handlers': {
        'file': {
            'class': 'logging.handlers.RotatingFileHandler',
            'filename': '/var/log/ics/django.log',
            'filters': ['redact_tokens'],  # ← apply filter
        },
    },
}

# logging_config.py
import re
class RedactTokensFilter(logging.Filter):
    def filter(self, record):
        record.msg = re.sub(
            r'(?i)(token|authorization)["\']?\s*[:=]\s*["\']?([a-f0-9]{40})',
            r'\1: [REDACTED]',
            str(record.msg)
        )
        return True
```

For higher security, implement token rotation on each login (see Part 5).

```python

    path('login/', TokenObtainPairView.as_view(), name='token_obtain_pair'),       # POST → access + refresh
```

### 5.4 Token Expiry Improvements (Can Add Now)

Even with Token auth, add a simple expiry field to prevent indefinite token lifetime:

```python
# api/models.py or signals
from django.utils import timezone
from datetime import timedelta

@receiver(post_save, sender=User)
def create_auth_token(sender, instance=None, created=False, **kwargs):
    if created:
        token = Token.objects.create(user=instance)
        # Add custom expiry: 90 days for MVP
        token.expires_at = timezone.now() + timedelta(days=90)
        token.save()

# Then in TokenAuthentication check:
def authenticate_token(key):
    try:
        token = Token.objects.get(key=key)
        if token.expires_at and timezone.now() > token.expires_at:
            raise AuthenticationFailed('Token has expired.')
        return (token.user, token)
    except Token.DoesNotExist:
        raise AuthenticationFailed('Invalid token.')
```

## Part 6 — Decision Log

| Decision | Rationale |
|----------|-----------|
| Token auth (not JWT) for MVP | Revocation is instant (DB delete). Web + internal API scope doesn't need JWT statelessness. Mobile deferred post-MVP. |
| Session auth for web (not Token) | Sessions are the browser-native security model. `HttpOnly` cookies are more secure than localStorage. CSRF token sent via header. |
| One token per user | Simplifies revocation. User can't have multiple active tokens (for MVP). Can add multi-device tokens later. |
| No token rotation (MVP) | Complexity deferred. Token lifetime is session lifetime (revoked on logout). Can add rotation in Phase 9. |
| No token expiry (MVP) | Login sets token; logout deletes it. No expiry clock needed. Can add `expires_at` field later (see Part 5). |
| Token in Authorization header (not cookie) | Separates web session (cookie) from API token (header). DRF expects Bearer tokens. Clearer intent. |
| Blacklist deferred | JWT requires blacklist; Token auth uses DB delete. Blacklist logic added when JWT migrates. |

## Part 8 — Appendix: Common Questions

### Q: "Should we use JWT now instead of Token?"

**A:** No. JWT complexity (expiry, rotation, blacklist, refresh) is unnecessary for the current scope. When the mobile app is built, the migration is straightforward. This decision is explicitly reviewable and reversible.

### Q: "What if a user logs in from multiple devices?"

**A:** Current implementation: one token per user (deleted on logout). Multi-device tokens are a Phase 8+ feature. For MVP, each device login revokes the previous token. Users can manage sessions via `/accounts/sessions/` view (post-MVP).

### Q: "How do I revoke a user's access immediately?"

**A:** Mark `user.is_active = False` and delete all their tokens:
```python
user.is_active = False
user.save()
Token.objects.filter(user=user).delete()
```
Session users: next request without valid session redirects to login.
API users: next request without token gets 401.

### Q: "Is Token auth secure for production?"

**A:** Yes, if served over HTTPS. Token is a random 40-char string; secrets are never logged or exposed in URLs. Only risk is token theft (solved by HTTPS + secure client storage).

### Q: "Why not use API keys instead of tokens?"

**A:** Tokens are API keys (Bearer scheme). The difference is semantics. `djangorestframework-simplejwt` JWTs are also Bearer tokens. All are equivalent from the client perspective; Token auth is simpler for this phase.
