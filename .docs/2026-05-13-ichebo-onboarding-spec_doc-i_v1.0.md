**_Every node in the Ichebo network is a legitimate KGS-compliant community by definition. This is structural, not aspirational. Onboarding is the mechanism that makes it so._**

**-- PHILOSOPHY**

**The Onboarding Philosophy**

Deliberate growth. Human approval. The passport office model.

## **1.1 Why Onboarding Is a Governance Feature**

Most digital platforms treat onboarding as a friction problem to be minimised. Self-service sign-up, instant access, no verification. Ichebo treats onboarding differently - as a governance function with theological and institutional significance.

The KGS principle is clear: communities are constituted, not self-declared. A Sceptre Community exists because it has been recognised within an apostolic network, placed under appropriate covering, assigned its position in the governance hierarchy, and equipped to function. These are not administrative steps. They are the acts by which a community comes into legitimate existence within the KGS.

**The governance principle**

No community is simply a name on a database. Every community that enters the Ichebo ecosystem has been verified as KGS-compliant, placed in the correct position in the network hierarchy, and issued a licence that represents their recognised status in the network. Growth is deliberate, not viral. Integrity of the network is structural.

## **1.2 The Passport Office Model**

The cloud is the passport office. The Desktop is the freedom that follows.

A community visits the passport office once - to prove who they are, to receive recognition of their status, and to be issued the document that enables them to operate freely. They do not return to the passport office every day. After that first visit, they operate independently with the authority their document confers.

The Ichebo Cloud onboarding process is this one-time visit. After it, the community operates fully offline on Ichebo Desktop. The licence key is their passport. The tenant record is their recognised status. The materialised path position is their place in the network.

## **1.3 The Deliberate Growth Model**

There is no self-service sign-up to the Ichebo ecosystem. There is no instant access. There is no freemium tier. This is not a limitation - it is a feature of the governance model.

| **What conventional SaaS onboarding looks like** | **What Ichebo onboarding looks like**                                                           |
| ------------------------------------------------ | ----------------------------------------------------------------------------------------------- |
| Fill a form online - get access immediately      | Application reviewed by a human with authority                                                  |
| Credit card determines access level              | KGS compliance determines access and position                                                   |
| Any organisation can sign up for any tier        | Tier is assigned by the approving authority based on the community's actual governance position |
| Network grows as fast as marketing allows        | Network grows as fast as KGS-compliant communities are constituted                              |
| Network integrity maintained by terms of service | Network integrity maintained by the onboarding structure itself                                 |

**-- KGS COMPLIANCE**

**What KGS Compliance Requires**

Apostolic covering. Proper constitution. Governance alignment. Network position.

## **2.1 KGS Compliance - Definition**

A KGS-compliant community is one that:

- Has apostolic covering - is under recognised apostolic authority within the network hierarchy
- Is properly constituted - has been formally established as a Sceptre Community or Church Node
- Has governance alignment - operates according to the KGS framework (seven pillars, formation pathways, service orders)
- Has a legitimate position in the network hierarchy - its tier and materialised path are correct and authorised
- Has a designated steward - a person with at least Level 3 competence responsible for the community's governance

## **2.2 The KGS Compliance Checklist**

The approving authority uses this checklist when reviewing an application. All five categories must be satisfied before approval.

**Category 1 - Apostolic Covering**

| **Requirement** | Compliant | Not compliant |
| --------------- | --------- | ------------- |

| **Identified apostolic authority** | Named senior leader with Level 4+ in the network confirms covering | No named apostolic authority. Community is operating independently. |
| ---------------------------------- | ------------------------------------------------------------------ | ------------------------------------------------------------------- |

| **Active relationship** | Apostolic leader is in regular contact with the community | Covering is nominal - no active relationship or accountability |
| ----------------------- | --------------------------------------------------------- | -------------------------------------------------------------- |

| **Authority is recognised in the network** | The apostolic leader is themselves recognised within the Ichebo network hierarchy | The apostolic leader is external to the network with no recognised status |
| ------------------------------------------ | --------------------------------------------------------------------------------- | ------------------------------------------------------------------------- |

**Category 2 - Community Constitution**

| **Requirement** | Compliant | Not compliant |
| --------------- | --------- | ------------- |

| **Formal establishment** | Community has been formally established - gathering regularly, members enrolled, leadership structure in place | A group of people who want to form a community but have not yet done so |
| ------------------------ | -------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |

| **Community tier** | Correct tier identified (church_node, church_collective, district, etc.) based on actual community scope | Tier claimed does not match actual community scope or authority |
| ------------------ | -------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------- |

| **Community steward identified** | A Level 3+ person is named as the responsible steward of this community | No steward identified. Community has no designated governance leader. |
| -------------------------------- | ----------------------------------------------------------------------- | --------------------------------------------------------------------- |

**Category 3 - Governance Alignment**

| **Requirement** | Compliant | Not compliant |
| --------------- | --------- | ------------- |

| **KGS awareness** | Community leadership demonstrates understanding of the KGS framework - seven pillars, formation pathways, service orders | No KGS knowledge. Community has not engaged with the framework. |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------- |

| **Formation commitment** | Community is committed to running formation pathways - induction, qualification programmes, competence tracking | No formation infrastructure. Community treats membership as affiliation not formation. |
| ------------------------ | --------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |

| **Governance practice** | Community has some form of governance structure in place - regular leadership meetings, accountability relationships, decision-making process | No governance structure. Leadership is informal with no accountability. |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |

**Category 4 - Network Position**

| **Requirement** | Compliant | Not compliant |
| --------------- | --------- | ------------- |

| **Correct parent tenant** | The community's position in the materialised path hierarchy is correct - their parent node exists and the relationship is recognised | Parent node does not exist or the relationship is claimed but not recognised |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------- |

| **Geography alignment** | Community location aligns with their claimed position in the geographic hierarchy | Community in Johannesburg cannot be placed under a Cape Town district node |
| ----------------------- | --------------------------------------------------------------------------------- | -------------------------------------------------------------------------- |

**Category 5 - Steward Verification**

| **Requirement** | Compliant | Not compliant |
| --------------- | --------- | ------------- |

| **Steward identity** | Named steward can be verified as a real person with an active relationship to the community | Steward cannot be verified or has no active relationship with the community |
| -------------------- | ------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |

| **Steward competence** | Steward has achieved or is working toward Level 3 competence | Steward has no formation history and no formation commitment |
| ---------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |

| **Steward account** | Steward has or will create an Ichebo Cloud account (the administrator account for Desktop activation) | Steward is unwilling to use the platform personally |
| ------------------- | ----------------------------------------------------------------------------------------------------- | --------------------------------------------------- |

## **2.3 Who Can Approve**

The approving authority must be a recognised leader within the Ichebo network with appropriate authority. The tier of the applying community determines the minimum level of the approving authority.

| **Community tier applying for** | **Minimum approving authority level**                                       |
| ------------------------------- | --------------------------------------------------------------------------- |
| church_node (Sceptre Community) | Level 4 - district steward or higher within whose scope the community falls |
| church_collective               | Level 4 - district or provincial steward                                    |
| district                        | Level 5 - apostolic steward with provincial or national authority           |
| provincial / national           | Level 5 - apostolic steward with continental or global authority            |
| First community in a new region | Level 5 - Paul Reuben's team (apostolic authority directly)                 |

**Paul Reuben's team - initial approver**

In the early growth phase of the Ichebo network, all applications will be reviewed and approved by Paul Reuben's team at Ichebo Christian Services. As the network grows and district and provincial stewards are established with sufficient authority, approval authority will be delegated to them. Delegation is itself a governance act - not an automatic process.

**-- THE FULL FLOW**

**Complete Onboarding Flow**

Nine steps. One direction. From application to offline operation.

## **3.1 The Nine-Step Onboarding Flow**

Every community that joins the Ichebo ecosystem passes through this flow. No shortcuts. No exceptions.

| **1** | **Application submitted** _(Community leadership)_<br><br>The community steward or an appointed representative submits an application via the Ichebo Cloud web interface (app.ichebo.org/apply). The application form collects: community name, location, tier claimed, apostolic covering (named authority), steward identity, brief description of community constitution, and contact details. |
| ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| **2** | **Initial review** _(Ichebo Cloud - automated)_<br><br>The system performs an automated initial check: is the claimed parent tenant valid? Does it exist in the network? Does the claimed tier make sense for the claimed geographic position? If the automated check fails, the applicant is notified with a specific reason and invited to resubmit. |
| ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |

| **3** | **Assignment to approving authority** _(Ichebo Cloud system)_<br><br>The application is assigned to the appropriate approving authority based on the community's claimed tier and geographic position. The approver is notified via email and in-app notification: "A new community application requires your review." |
| ----- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| **4** | **KGS compliance review** _(Approving authority - human in the loop)_<br><br>The approver reviews the application against the five compliance categories (Part 2.2). They may: contact the applicant for additional information, contact the named apostolic authority to verify covering, contact references provided. This step requires genuine human judgment - it cannot be automated. The five compliance categories are a guide, not a checklist that replaces discernment. |
| ----- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| **5** | **Decision** _(Approving authority - human)_<br><br>The approver makes one of three decisions: Approve (community is KGS-compliant, proceed to onboarding), Hold (more information required - specific questions sent to applicant), or Reject (community does not meet KGS compliance requirements - reason provided). |
| ----- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| **6** | **Tenant creation** _(Ichebo Cloud - automated on approval)_<br><br>On approval: the Tenant record is created with the correct path, tier, and parent. The steward's administrator account is created or linked. A UserPermission is granted at Level 3 (or existing level) for the steward in the new tenant. The community is now visible in the network hierarchy. |
| ----- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| **7** | **Licence key generation and issuance** _(Ichebo Cloud - automated)_<br><br>A unique licence key is generated and issued to the community steward by email. The key is a cryptographically random string tied to the tenant UUID and the steward's account. It is not reusable - each key activates one Desktop installation for one community. |
| ----- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| **8** | **Desktop activation** _(Community steward)_<br><br>The steward downloads Ichebo Desktop and enters the licence key on first run. The Desktop calls POST /api/sync/validate-licence/ with the key. The cloud confirms the key, returns the tenant identity, and initiates the initial sync payload download. |
| ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |

| **9** | **Initial sync and independent operation** _(Ichebo Desktop + Cloud)_<br><br>The Desktop receives the initial sync payload (tenant settings, member roster, last 90 days of activities, governance snapshot). After the initial sync completes, the community is fully operational offline. They never need to return to the passport office. |
| ----- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

## **4.2 The Materialised Path - Immutable After Creation**

**Non-negotiable**

The materialised path is set at tenant creation and is immutable without a formal process. A community cannot move itself up or down the hierarchy. If a community's governance position changes - a district node is elevated to provincial status, or a community merges with another - this requires a formal process with approving authority approval and a data migration.

The path structure follows the geographic and governance hierarchy:

// Materialised path examples

// A church node in Pretoria North, Gauteng, South Africa

/global/africa/southafrica/gauteng/pretoria-north/sceptre-community-abc/

// A district node for Gauteng

/global/africa/southafrica/gauteng/

// A national collective for South Africa

/global/africa/southafrica/

// Scope query - find all records visible to the Gauteng district steward:

WHERE tenant_path LIKE '/global/africa/southafrica/gauteng/%'

// Find all communities under the Pretoria North node:

WHERE tenant_path LIKE '/global/africa/southafrica/gauteng/pretoria-north/%'

AND tier = 'church_node'

## **4.3 Initial UserPermission**

On tenant creation, the steward is granted a UserPermission in the new tenant:

}

**-- LICENCE KEY**

**Generation, Issuance, and Validation**

One key. One community. One Desktop installation.

## **5.1 Licence Key Generation**

The licence key is generated automatically when the tenant is created. It is:

- Cryptographically random - generated with a secure random number generator, not sequential or guessable
- Tied to the tenant UUID and the steward's user account UUID
- Not reusable - each key activates exactly one Desktop installation
- Not time-limited by default - a community should not lose access because they delayed installation
- Revocable by the approving authority - if a community's status changes, their licence can be revoked

// Licence key structure (internal - not exposed to users in this format)

LicenceKey {

key: "ICH-" + random_urlsafe_string(32), // e.g. "ICH-xK9mP2wQrYh4nL7jBvF8cD3tA6sE1gZ"

tenant_id: uuid, // the community this key is for

steward_id: uuid, // the steward who should activate it

status: "active", // active | revoked | activated

issued_at: ISO-8601,

activated_at: null, // set when Desktop first uses the key

revoked_at: null, // set if revoked

revoked_by: null // uuid of the authority who revoked

}

// The user sees only: "ICH-xK9mP2wQrYh4nL7jBvF8cD3tA6sE1gZ"

// They enter this in Ichebo Desktop on first run

## **5.2 Licence Key Issuance**

The licence key is delivered to the community steward by two channels simultaneously:

- Email to the steward's registered email address - sent via Brevo. Subject: "Your Ichebo Desktop licence key - {Community name}"
- In-app notification on the steward's Ichebo Cloud account - if they have already registered

The email contains:

- The licence key (displayed prominently, easy to copy)
- A link to download Ichebo Desktop for their operating system
- Step-by-step installation and activation instructions
- A support contact if the activation fails

## **5.3 Desktop Licence Validation**

When the steward enters the licence key on first run, Ichebo Desktop calls the licence validation endpoint:

// POST /api/sync/validate-licence/

// Called by Ichebo Desktop on first run when the steward enters their licence key

Request:

{

"licence_key": "ICH-xK9mP2wQrYh4nL7jBvF8cD3tA6sE1gZ",

"device_id": "uuid-generated-on-first-run"

}

Response - success (200 OK):

{

"valid": true,

"tenant_id": "uuid",

"tenant_name": "Sceptre Community Pretoria North",

"tenant_path": "/global/africa/southafrica/gauteng/pretoria-north/sceptre-community-abc/",

"steward_id": "uuid",

"expires_at": null // null = no expiry

}

Response - failure (403 Forbidden):

{

"valid": false,

"reason": "revoked" // "revoked" | "already_activated" | "not_found" | "expired"

}

// On success:

// 1. Desktop stores tenant_id, tenant_path in SQLite settings

// 2. Desktop stores device_id in SQLite settings (permanent)

// 3. Licence key status updated to "activated" on cloud

// 4. Desktop proceeds to initial sync

## **5.4 Periodic Licence Validation**

Ichebo Desktop validates the licence periodically to confirm the community's status has not changed. This is not a connectivity requirement - it is a governance check.

| **Validation event**     | **Frequency**                                                                   |
| ------------------------ | ------------------------------------------------------------------------------- |
| On first run             | Required - activation cannot proceed without successful validation              |
| On app open (if online)  | Monthly - the Desktop checks once per calendar month when internet is available |
| Manual (in Sync section) | On demand - steward can trigger validation from the Sync section                |

If monthly validation fails (licence revoked or community status changed):

- Sync Engine transitions to Blocked state
- Status bar shows: "Sync blocked - contact your administrator"
- All locally stored data remains accessible - the community's data is never deleted
- The community can continue operating offline with their existing data - they cannot sync new data to the cloud
- The steward contacts their approving authority to resolve the status

**-- INITIAL SYNC**

**The Initial Sync Payload**

What the cloud delivers to Desktop on first activation.

## **6.1 Initial Sync Payload**

When the licence validates successfully, the Desktop immediately begins the initial sync. This downloads the community's data from the cloud and populates the local SQLite database.

**Performance target**

The initial sync must complete in under 60 seconds on a 5 Mbps connection for a community of 500 members. If the payload would exceed this, paginate. Never block the first-run experience. The steward should be operational within 5 minutes of entering their licence key.

## **6.2 What the Initial Sync Contains**

| **Data category**          | **What is included**                                                                                                                                   |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Tenant settings            | name, tier, path, area_of_operation, location, settings fields. The community's identity on the device.                                                |
| Full member roster         | All active UserPermission records for this tenant. Names, competence levels, service orders, shepherd assignments. Every current member.               |
| Last 90 days of activities | Activity records (gatherings, service events, tasks, campaigns) from the last 90 days. Attendance records. The community's recent operational history. |
| Governance snapshot        | Active mandate, reference library, and keys records from the Handbook. Read-only. The governance framework the community operates under.               |
| Steward account            | The steward's own User record, UserProfile, and UserPermissions. Their formation history. Their formation profile.                                     |

## **6.3 What the Initial Sync Does NOT Contain**

| **Not included**                | **Why**                                                                                                    |
| ------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| Bible verse data                | Bundled with the installer - 30MB SQLite file (KJV, ASV, WEB). Available from minute one without download. |
| Activity history beyond 90 days | Available on demand if the steward scrolls back. Not in the initial payload for performance reasons.       |
| Learn programme content         | Not in Desktop MVP scope. Synced on demand in Desktop Phase 2.                                             |
| Other communities' data         | Each Desktop installation is scoped to one community. Cross-community data is cloud-only.                  |
| Handbook governance history     | Version history of governance documents is cloud-only. Desktop receives only the current active version.   |

## **6.4 Initial Sync Endpoint**

The initial sync uses the standard pull endpoint with a special since=epoch parameter to retrieve all data:

// Initial sync - retrieve all data (no since= filter)

GET /api/sync/pull/

?device_id={uuid}

&tenant_id={uuid}

&initial=true // signals this is an initial sync - no since filter

// The cloud returns paginated responses.

// Desktop continues paginating until has_more = false.

// Progress displayed to the steward on Screen 3 of the first-run wizard.

// After initial sync completes:

// 1. Desktop sets LastSyncedAt to the retrieved_at timestamp

// 2. Desktop transitions to Synced state

// 3. Steward lands on the Home section

// 4. The community is fully operational offline from this point

**-- LIFECYCLE**

**Community Lifecycle Management**

Active. Suspended. Revoked. Status changes require authority.

## **7.1 Tenant Status**

| **Status** | **What it means**                                                                                                                      |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| active     | Community is operating. Licence is valid. Sync works. All platform features available.                                                 |
| pending    | Application submitted but not yet approved. Account created, induction access only.                                                    |
| suspended  | Community is temporarily inactive - licence validation will return Blocked. Data preserved. Can be reactivated by approving authority. |
| revoked    | Licence permanently revoked. Sync permanently blocked. Community is removed from the network. Data preserved locally.                  |

## **7.2 Status Change Process**

Community status changes require authority and a documented reason. They are governance acts, not administrative actions.

| **Status change**                  | **Authority required**                                                           |
| ---------------------------------- | -------------------------------------------------------------------------------- |
| pending → active (approval)        | Approving authority - Level 4+ (see Part 2.3)                                    |
| active → suspended                 | Level 4+ within whose scope the community falls. Reason documented.              |
| suspended → active (reinstatement) | Level 4+ - same authority who suspended, or higher. Compliance reassessed.       |
| active or suspended → revoked      | Level 5 - apostolic authority. Final act. Reason documented. Cannot be reversed. |

## **7.3 What Happens When a Community is Suspended or Revoked**

**On suspension**

- Licence validation returns: { valid: false, reason: "suspended" }
- Sync Engine transitions to Blocked state on next validation
- Status bar shows: "Sync blocked - contact your administrator"
- All locally stored data remains accessible and readable
- No new data can be pushed to the cloud
- Community can continue operating locally with existing data
- Approving authority notifies the community steward with the reason and any conditions for reinstatement

**On revocation**

- Licence validation returns: { valid: false, reason: "revoked" }
- Sync Engine permanently blocked
- Community removed from the network hierarchy - tenant path is deactivated
- All locally stored data remains on the device - never deleted
- The community's data is not accessible to other network members
- The decision is final. The community would need to reapply as a new community to re-enter the network.

**Data preservation principle**

In no scenario is community data deleted by the platform. A community's members, activities, governance records, and formation history belong to the community. Even if their network status changes, their data remains on their device. The platform is a steward of data, not its owner.

# **Part 10 - Build Sequence and Open Items**

## **10.1 Current Production State**

The current production system (Version 2) has the following onboarding infrastructure in place:

- Tenant model with materialised path and tier fields - fully built
- TenantInvitation model for inviting members to existing tenants - fully built
- Steward dashboard with tenant creation form (Level 3+) - fully built
- Six Agency Tenants seeded as constitutional records - fully built
- Induction tenant as system singleton - fully built

What the current system does NOT have:

- CommunityApplication model and workflow - communities are currently created directly by stewards with no formal application process
- LicenceKey model and generation - no Desktop activation infrastructure yet
- POST /api/sync/validate-licence/ endpoint - not yet built
- Network tree view for approvers - not yet built
- Application review panel - not yet built

## **10.2 Build Phases**

| **Phase**                               | **What it builds**                                                                                |

| I.1 - CommunityApplication model        | Application model, form, submission endpoint. Public-facing application form at ichebo.org/apply. |

## **10.3 Entry Requirement**

**Gate**

The onboarding system (Phases I.1-I.5) must be complete before Ichebo Desktop is released to the public. A community cannot activate Desktop without a licence key. A licence key cannot be issued without an approved application. The onboarding system is the gate that makes the network's integrity structural.

## **10.4 Open Items**

- Licence key expiry policy - current decision: no expiry. Should there be an annual renewal for governance accountability? To be decided.
- Multiple Desktop installations - can one community activate Desktop on multiple PCs? Current: one key per community. Second key would require a separate process.
- Community self-update - can a community update their own details (new steward, updated location) without a full reapplication? To be designed.
- Application appeal process - currently a rejected community can reapply after 30 days. Formal appeal process to be designed.
- Delegated approval authority - when and how is approval authority delegated from Paul Reuben's team to district/provincial stewards? To be governed.
