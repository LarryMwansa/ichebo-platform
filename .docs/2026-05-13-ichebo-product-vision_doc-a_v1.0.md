**What Ichebo Is**

The digital expression of the Kingdom Governance System - a complete operating system for apostolic communities.

KGS is the framework. The Sceptre Community Programme is the movement. Ichebo is the platform. These are three different things at three different layers. The platform serves the programme. The programme is ordered by the framework. Never conflate the three.

## **1.2 What Ichebo Is Not**

Ichebo is not a church management system. It is not a generic SaaS platform adapted for churches. It is not a product looking for a market. It is the digital expression of a complete institutional framework that was developed before the platform existed.

The distinction matters because it determines every product decision. Features are not added because users request them or because competitors offer them. Features are added because the KGS requires them. The KGS is the product requirements document.

## **1.3 The Foundational Problem Being Solved**

The digital gap in faith-based organisations is not a technology adoption problem. It is a governance problem.

Most existing church management software is designed for the Western congregational model: a single pastor, a single congregation, a weekly gathering, and a membership database. It does not account for the structural reality of apostolic networks, multi-tiered governance hierarchies, community-based discipleship systems, or the relational complexity of networked ministry organisations operating across multiple geographies.

The consequences of operational fragmentation compound over time:

- Leadership accountability becomes difficult to maintain across the network
- Discipleship and formation pathways are inconsistently applied
- Community data is lost or inaccessible between leadership transitions
- Financial stewardship is weakened by poor record-keeping
- Mission activity is uncoordinated across communities
- Demonstrable social impact - increasingly required by institutional funders - is impossible to evidence without coherent operational records

**_The problem is not that faith-based organisations lack the will to modernise. It is that no platform has been built that genuinely fits how they are structured and how they actually operate._**

**-- PHILOSOPHY**

**The Local-First Commitment**

The device is the primary computer. The cloud is coordination. A community's governance data belongs on a device the community controls.

## **2.1 What Local-First Means**

Local-first is not a technical feature. It is a philosophical commitment about who owns data and who depends on whom.

In a cloud-first architecture: the cloud owns the data. If the cloud is unavailable - because of a server failure, an internet outage, a power cut, or because the vendor raises prices beyond reach - the community cannot operate. The community is dependent.

In a local-first architecture: the device owns the data. The cloud receives a copy. If the cloud is unavailable, the community continues operating without interruption. The cloud depends on the community - not the other way around.

| **Principle**          | **What it means in practice for a Sceptre Community**                                                                                                    |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Device is primary      | All data is written to the local device first. The cloud receives a synchronised copy - it does not hold the original.                                   |
| Cloud is coordination  | The cloud connects communities to each other, gives apostolic leadership network visibility, and enforces KGS compliance. It does not run the community. |
| Offline is the default | Every feature must work without internet. Connectivity is an enhancement, not a dependency.                                                              |
| Sync is deliberate     | Data moves from device to cloud on sync - not on every write. The community controls when their data leaves the device.                                  |
| Data sovereignty       | The community owns their member records, governance documents, and formation history on their own device. The cloud sees it only when they sync.         |

## **2.2 Why Local-First for Africa**

The local-first commitment is not an abstract architectural principle for Ichebo. It is a direct response to the primary deployment context.

Sub-Saharan Africa is the fastest-growing Christian population in the world and the primary launch market for Ichebo. It is also a region where:

- Load-shedding can cut power and internet for 4-12 hours a day in some areas
- Mobile data is expensive relative to income, making continuous cloud connectivity a significant cost
- Rural communities may have no reliable broadband at all
- Second-hand and older devices are the norm - not the latest hardware
- Trust in cloud service providers is reasonably lower than in Western markets

A community in Lusaka, Harare, or Lagos should not need an internet connection to know who their members are, record their gatherings, or track their formation progress. Ichebo must work in the conditions that actually exist - not the conditions that would be convenient for the platform.

**The test**

If the internet goes down permanently in a region for six months, Ichebo Desktop must work perfectly and lose nothing of operational value. The cloud was always the optional layer.

## **2.3 Data Sovereignty as a Kingdom Value**

Data sovereignty is not only a practical concern - it is a values alignment with the KGS.

The KGS places authority at the level of the local community, ordered upward through the network hierarchy. The Sceptre Community is not a branch office of a central institution - it is a constituted community with its own gifts, calling, and responsibilities before God. Its data - its members, its governance records, its formation history - belongs to the community, stewarded within the network, not owned by a technology vendor.

Local-first architecture is the technical expression of this value. The community's data lives on a device the community controls. The cloud serves the network. The community is not a data subject of the platform - it is the platform's principal.

## **3.3 The Cloud as Passport Office**

Before Ichebo Desktop activates for a community, that community must be KGS-verified. The cloud's role in this process:

- Verify KGS compliance - apostolic covering, tier, network position
- Set up the tenant correctly in the materialized path hierarchy
- Issue the Desktop installation with a verified identity and licence key
- Provide network leadership with cross-community visibility
- Coordinate sync between Desktop nodes across the network

After this onboarding handshake, the Desktop operates independently. The community goes once to the passport office to prove who they are. After that they operate freely without returning. The licence key is their passport.

**The deliberate growth model**

Every community that enters the Ichebo ecosystem requires human approval by an authorised leader before a licence key is issued. This is not a limitation - it is a governance feature. Growth is deliberate, not viral. Every node in the network is a legitimate KGS-compliant community by definition.

**-- ECOSYSTEM**

**One Architecture - Three Products**

The domain engines are the shared foundation. Every product surface calls them. None owns them.

## **4.1 The Ecosystem Map**

The Ichebo ecosystem is organised in four layers. Every product surface sits on top of the same foundation.

| **CLIENT SURFACES**                                                                                         |
| ----------------------------------------------------------------------------------------------------------- |
| Ichebo Desktop (Flutter) \| Ichebo Mobile (Flutter) \| Ichebo Web (Django + HTMX)                           |
| **FOUNDATION ENGINES (Go modules - portable, embeddable)**                                                  |
| Records Engine \| Activity Engine \| Relationships Engine \| Bible Engine \| Calendar Engine \| Sync Engine |
| **CLOUD COORDINATION (Django 4.2 + DRF + PostgreSQL on Hetzner)**                                           |
| KGS Compliance \| Tenant Management \| Network Visibility \| Sync Coordination \| Licence Issuance          |

## **4.2 The Sync Engine - The Strategic Moat**

The Sync Engine is the component that makes the local-first architecture possible. It is the most strategically significant piece of engineering in the ecosystem.

It is a standalone Go binary - embedded in Ichebo Desktop, running as a background daemon. It tracks every local change, pushes changes to the cloud when online, pulls cloud changes down, and resolves conflicts by a defined set of rules.

It is owned entirely by Ichebo:

- Not a third-party library - cannot be deprecated by a vendor
- Not a SaaS dependency - cannot be price-increased away
- Not a cloud feature - works embedded on every device
- Potentially publishable as the Ichebo Sync Protocol - a standard for KGS-compliant community data synchronisation

**The moat**

Whoever controls the sync layer controls the network. The Sync Engine is not a feature. It is the competitive infrastructure that cannot be replicated by a competitor buying an off-the-shelf component.

## **4.3 Django's Role - Clarified**

Django is the web layer. It is not the domain logic layer.

Django is excellent at what it was designed to do: handle HTTP requests, serve templates, manage authentication, talk to a PostgreSQL database. The Version 1 and 2 Django build proved the domain - it showed what the platform needed to be and established the data contract that governs the entire ecosystem.

The Go engines are the domain logic layer - portable, embeddable, language-agnostic in specification, available to every product surface. Django calls the engines. Django does not own the logic.

The three-phase separation plan:

- Phase 1 (current): business rules extracted into Python modules that Django calls - not Django doing the logic itself
- Phase 2 (Ichebo Desktop build): Go engines built. Cloud Django begins calling engine specifications. Django becomes a thin adapter.
- Phase 3 (as products mature): services extracted as each domain's complexity demands. API Gateway formalised. Clients do not know which service answers.

**-- GROWTH**

**The Growth Model**

Deliberate, not viral. Every node in the network is a legitimate KGS-compliant community by definition.

## **7.1 Deliberate Growth**

The Ichebo growth model is structurally different from conventional SaaS growth. There is no self-service sign-up. There is no viral loop. There is no freemium tier with upsell pressure.

Every community that enters the Ichebo ecosystem passes through KGS onboarding - a human-in-the-loop process where an authorised leader verifies that the community is genuinely KGS-compliant before a licence is issued. This is not a limitation on growth. It is a governance feature that ensures the integrity of the network.

The onboarding flow:

- Community applies via Ichebo Cloud
- KGS compliance verified - apostolic covering, tier, network position confirmed
- Human approval by authorised leader - not a checkbox, not a form
- Tenant created in cloud - position in network hierarchy locked
- Desktop licence key generated and issued
- Community installs Ichebo Desktop, activates, syncs initial data
- Community operates fully offline from this point

The result: every node in the Ichebo network is a legitimate KGS-compliant community by definition. The network's integrity is structural, not aspirational.

## **7.2 The Network Effect**

Every community that activates Ichebo Desktop becomes a node. When they sync to the cloud, apostolic leadership has visibility across every community simultaneously. The Desktop is not a downgrade from the cloud - it is the data collection layer that makes the cloud's network visibility meaningful.

As the network grows:

- Apostolic leadership gains real-time visibility across the full network
- Formation data aggregates upward - leadership can see where the network is in formation
- Mission coordination becomes possible across communities that were previously operating in isolation
- The Sceptre Community Programme becomes measurable and evidenceable for funders and partners

**-- TECHNOLOGY**

**Technology Stack - Principled Choices**

Each technology chosen for a specific reason. No defaults, no fashions, no vendor lock-in.

## **8.1 Stack by Layer**

| **Layer**                | **Technology**          | **Why**                                                                                                                                                     |
| ------------------------ | ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Desktop + Mobile UI      | Flutter / Dart          | One codebase - Windows, macOS, Linux, Android, iOS. Native rendering per platform. Strong offline/SQLite support. Growing in African developer communities. |
| Domain Engines           | Go                      | Fast, concurrent, single binary, embeddable, no runtime dependencies. Purpose-built for portable domain logic.                                              |
| Cloud Web Layer          | Python / Django 4.2 LTS | Stable, well-documented, proven for this domain. Web layer only - not domain logic owner.                                                                   |
| Cloud API                | Django REST Framework   | Mobile and Desktop sync target. Retained as the canonical service layer.                                                                                    |
| Local database           | SQLite (WAL mode)       | Embedded, zero config, no server process, battle-tested for mobile and desktop.                                                                             |
| Cloud database           | PostgreSQL              | Relational, robust, running on Hetzner VPS.                                                                                                                 |
| Object storage (interim) | MinIO on VPS (80GB HDD) | S3-compatible, ample for current stage. Bucket: ics-media. Migration to Hetzner Object Storage when Ichebo Media requires CDN delivery.                     |
| Video (future)           | Go + FFmpeg + HLS       | Purpose-built video pipeline. Not Django. Not an embedded player only.                                                                                      |

## **8.2 What We Deliberately Are Not**

| **Not this**              | **Why**                                                                                                       |
| ------------------------- | ------------------------------------------------------------------------------------------------------------- |
| React / Vue on the web    | Django templates + HTMX is the locked web UI layer. Simpler, server-rendered, no JS framework complexity.     |
| Docker in Version 2       | Nginx + Gunicorn + systemd is the proven production stack. Docker is a Version 3 decision.                    |
| LLM-based Paraclete (MVP) | Rule-based Python in MVP. Pure function, no API cost, no hallucination risk. LLM is a Layer 10 consideration. |

| Vendor sync engine        | Ichebo owns the sync layer. The moat cannot be bought or deprecated away.                                     |
