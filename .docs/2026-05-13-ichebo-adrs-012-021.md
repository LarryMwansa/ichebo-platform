**ADR-012**

**Apostolic Command Shell as Canonical Desktop Governance Interface**

| **Field** | **Value**                            |
| --------- | ------------------------------------ |
| Number    | ADR-012                              |
| Status    | APPROVED                             |
| Date      | May 2026                             |

## **Context**

The Ichebo platform master roadmap described the desktop workspace as "Django templates + HTMX for Level 3+ operators." During the Version 2 build, the desktop workspace evolved significantly beyond this description into a named, designed, architecturally specific product: the Apostolic Command Shell.

The shell was built, iterated on, and refined through a dedicated refinements methodology. By the completion of Version 2, the shell was functionally complete with all templates in place, the four-column grid architecture working, and the dual-shell switching between Stage Mode and Mobile Mode operational.

## **Decision**

**DECISION**

_The Apostolic Command Shell is the canonical desktop governance interface for the Ichebo platform. It is a named, designed product - not a generic Django web interface. Its architecture is locked as described in this ADR and in DESIGN.md (see ADR-013)._

## **Architecture**

The Apostolic Command Shell implements a four-column grid architecture:

| **Zone**                   | **Description**                                                                                                                                               |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Column 1 - Primary Sidebar | 72px ink dark nav rail. Global app switcher with icons. Tenancy Hub pinned to bottom. Active state: 3px red left rule.                                        |
| Column 2 - Context Bar     | 240px. App-specific navigation, actions, folder structure, editorial controls. Styled uniformly using the Governance app button standard.                     |
| Column 3 - The Stage       | Flexible width. The sovereign canvas - editorial markdown authorship, record rendering, data display. Stone surface. Playfair Display for all stage headings. |
| Column 4 - Options Bar     | 300px stone surface. Tabbed sidecar - record metadata, links, relationships, forms, quick-access tools relevant to the active stage window.                   |

## **Navigation Structure**

Primary sidebar navigation order (top to bottom):

- Dashboard (Command Centre)
- The Desk
- Governance
- Community
- Formation
- Activity
- Bible Reader
- Video and Broadcast

Bottom-pinned (separated by visual divider):

- Tenancy Hub

## **Level Gating**

The Apostolic Command Shell activates when: user is authenticated AND competence_level >= 3 AND viewport >= 1024px. The .ws-active class is added to the body element by the inline script in base.html. Users at Level 0-2 never see the workspace shell - they see the Mobile Mode rendering path exclusively.

## **Consequences**

- All desktop template work references the Apostolic Command Shell architecture. No desktop template may be built outside this shell structure.
- The four column names - Primary Sidebar, Context Bar, Stage, Options Bar - are the canonical names. No renaming without a new ADR.
- The Governance app context bar button style is the standard for all context bar styling across all apps.
- DESIGN.md governs all visual decisions within the shell - see ADR-013.
- The shell is a Version 2 deliverable. It is production-ready. Further refinements are minor polish, not structural changes.

## **Context**

The Ichebo platform operates across three surfaces: the marketing site (ichebo.online), the desktop workspace (app.ichebo.org running the Apostolic Command Shell), and the Flutter mobile app. During the Version 2 build, a comprehensive design system was produced and locked in DESIGN.md, with design-preview.html as its canonical living reference implementation containing all visual assets, components, and design tokens rendered as working HTML/CSS.

Prior to this ADR, DESIGN.md existed as a file but was not formally recognised in the data contract or roadmap as an authoritative governance document. This ADR formalises its authority.

## **Decision**

**DECISION**

_DESIGN.md is the locked design system authority for all Ichebo surfaces. design-preview.html is its canonical visual reference implementation. Both are governed together. No design decision that contradicts either document may be implemented without first amending both documents through a formal design amendment process._

## **What DESIGN.md Governs**

| **Domain**          | **Authority**                                                                                                                                                                                                                        |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Typography          | Playfair Display (display/hero) + Inter (UI/body) + Fira Code (monospace). Type scale, weight, tracking rules. No other typefaces without amendment.                                                                                 |
| Colour              | Full brand palette: Primary Red #AF3236, Secondary Blue #185ABC, Ink #0E0E0E, Stone #F5F3F0. Level colours, semantic colours, dark mode strategy. Pink (#FFC0CB or similar) is never an accent colour - all pink instances are bugs. |

## **What design-preview.html Governs**

design-preview.html is the living visual reference for the design system. It contains:

- All design tokens rendered as visible colour swatches, spacing blocks, and type specimens
- All signature patterns implemented in working HTML/CSS
- All shared components rendered in their correct states
- Both light mode and dark mode implementations
- The complete CSS variable set that all templates and the Flutter theme must reference

design-preview.html is the ground truth for how design tokens translate to visual output. Where DESIGN.md describes intent, design-preview.html shows implementation. Both are authoritative. In cases of ambiguity, design-preview.html is the tiebreaker.

## **Dark Mode**

Dark mode is not optional on any surface. It must be built from day one on every new surface and feature. The dark mode strategy is defined in DESIGN.md: surface shift (--bg to #0E0E0E, --card to #1A1A1A), text shift, border shift, semantic colour de-saturation. The primary red #AF3236 never changes in dark mode - it is a fixed brand anchor.

## **Amendment Process**

Amending the design system requires: (1) a written rationale for the change, (2) updating DESIGN.md, (3) updating design-preview.html to reflect the change visually, (4) a note in the decisions log at the bottom of DESIGN.md with date and rationale. No implementation of a design change before all three steps are complete.

## **Consequences**

- Every Django template, CSS file, and Flutter widget produced for Ichebo references the DESIGN.md token system.
- Pink accent colours (#FFC0CB or similar) appearing anywhere in the platform are bugs - fix immediately.
- Playfair Display is used exclusively for page titles, hero headings, stat display numbers. Never for navigation, buttons, labels, or body text.
- The Ghost Watermark, Label Tag, and Left Red Rule are the primary visual identifiers of the platform. They must appear consistently.
- Flutter tokens.dart and app_theme.dart are derived from DESIGN.md - they must be kept in sync.

**ADR-014**

**The Desk as Canonical Governance Authorship Surface**

| **Field** | **Value**                            |
| --------- | ------------------------------------ |
| Number    | ADR-014                              |
| Status    | APPROVED                             |
| Date      | May 2026                             |

## **Context**

During the Version 2 build of the Apostolic Command Shell, The Desk emerged as a distinct and significant component - an editorial governance authorship environment within the shell. The Desk was not described in the original roadmap; it emerged from the build as the natural answer to the question of where governance content is created and edited.

The Desk proved to be a revelation point in the platform's development. It demonstrated conclusively that: (1) governance authorship requires a dedicated editorial environment, not a generic form; (2) the Handbook cannot adequately be modelled as a tenant because its authorship needs exceed what a tenant can provide; (3) the Options Bar as a metadata sidecar is the correct pattern for record properties, links, and relationships.

## **Decision**

**DECISION**

_The Desk is the canonical governance authorship surface within the Apostolic Command Shell. It is a named, designed component with specific responsibilities. It is the proof of concept that directly validates the Ichebo Handbook as a standalone product (see ADR-020)._

## **What The Desk Is**

The Desk occupies the Stage (Column 3) of the Apostolic Command Shell when the user is in authorship mode. Its specific responsibilities:

- Markdown authorship canvas - the editorial writing surface for all governance records
- Record routing - record types selected in the Context Bar open their forms and editing surfaces in the Stage
- Markdown live preview - Obsidian-style edit/preview mode within the writing canvas
- Formatting toolbar - document-centric formatting tools positioned within the stage
- Record type selector - Context Bar houses record family and type navigation with collapsible accordion groups

## **The Desk and the Options Bar**

The Options Bar (Column 4) serves The Desk as:

- Properties pane - record type, metadata, author, version history
- Links and relationships - HRS relationship viewer, linked records
- Bible tool - scripture reference and linking tool for governance authorship (primary use case within The Desk)
- Quick-access forms - small forms relevant to the active record

## **What The Desk Proved**

The Desk demonstrated three things that are now locked as architectural directions:

- Governance authorship is a distinct discipline that requires its own environment. The generic records list view is for browsing. The Desk is for creating and editing. These are different cognitive modes and deserve different interfaces.
- The Handbook's authorship needs - HRS methodology, scripture mapping, mandate versioning, prophetic key management - exceed what can be provided within a tenant context. The Desk as a component shows the shape of what the Handbook product will be: an entire workspace dedicated to governance authorship at the network level.
- The Bible as a tool within The Desk (not as a standalone app in this context) shows the correct separation of the Bible Engine from the Handbook's HRS scripture module. The Bible provides text. The Desk provides the authorship context. The Handbook will provide the HRS relationship layer.

## **Consequences**

- The Desk is a Version 2 deliverable. It exists in production. Future work is refinement, not construction.
- All governance record creation and editing routes through The Desk. No new full-page forms for governance records.
- The Desk validates ADR-020 (Ichebo Handbook as standalone product). The design pattern of The Desk informs the Handbook product design.
- The writing canvas max-width is 680px centred in the Stage for comfortable reading measure.
- The toolbar height is 44px, distinct background from stage, border-bottom separator.

**ADR-015**

**Dual-Shell Rendering Architecture - Stage Mode and Mobile Mode**

| **Field** | **Value**                            |
| --------- | ------------------------------------ |
| Number    | ADR-015                              |
| Status    | APPROVED                             |
| Date      | May 2026                             |
| Authors   | Chizola (domain); Claude (technical) |

## **Context**

The Ichebo platform serves two fundamentally different user contexts from the same Django codebase: Level 3+ stewards operating on desktop (the governance and operations context) and all users on mobile devices (the daily community and formation context). During the Version 2 build, a dual-shell rendering architecture was developed to serve both contexts elegantly from a single template inheritance tree.

The prior documentation described mobile behaviour as "graceful degradation below 768px." This was an inadequate characterisation. The Mobile Mode UI is not a degraded desktop - it is a purpose-built mobile interface that is excellent in its own right and is the primary daily interface for the majority of Ichebo users.

## **Decision**

**DECISION**

_Stage Mode and Mobile Mode are two first-class, purpose-built rendering paths served from the same Django template codebase. Neither is a fallback or degradation of the other. Each is designed for its context and canvas. The Mobile Mode UI is the mainstay for mobile users. The Stage Mode is the exclusive interface for Level 3+ stewards on desktop._

## **How the Dual-Shell Works**

base.html implements the dual-shell pattern:

- Both blocks defined: {% block content %} (Mobile Mode) and {% block ws_content %} (Stage Mode)
- JS switching: inline script on base.html toggles .ws-active on &lt;body&gt; at viewport >= 1024px AND competence_level >= 3
- CSS switching: workspace.css hides/shows based on .ws-active class

| **Block**              | **Who sees it**                                                                                                        |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| {% block content %}    | All users on mobile viewport. Level 0-2 on any viewport. The Mobile Mode shell with bottom navigation and top app bar. |
| {% block ws_content %} | Level 3+ users on viewport >= 1024px exclusively. The Apostolic Command Shell four-column grid.                        |

## **Mobile Mode - Characterisation**

Mobile Mode is not a fallback. It is:

- The primary interface for Level 0-2 members - their only interface
- The primary daily interface for Level 3+ members using Ichebo on their phones
- A designed, complete mobile experience with its own navigation shell, component set, and interaction patterns
- The direct predecessor of the Flutter mobile app - the mobile UI and the Flutter app bear intentional resemblance in structure and visual language
- PWA-ready - the dual-shell architecture positions both surfaces for PWA manifest and service worker support

## **Stage Mode - Characterisation**

Stage Mode is not the "real" interface with mobile as its lesser version. It is:

- Exclusively for Level 3+ stewards in a governance and operations context
- Designed for information density - more data per viewport, compact spacing
- The Apostolic Command Shell as defined in ADR-012
- Inappropriate for seekers and new disciples - the governance density is not accessible to users still in formation

## **Template Architecture**

workspace_shell.html is the root template for Stage Mode. It carries {% block content %} to support mobile fallback rendering for any template that extends it. All seven major app templates implement both blocks - Stage Mode content in {% block ws_content %} and Mobile Mode content in {% block content %}.

## **Consequences**

- The description "graceful degradation" is retired. Mobile Mode is a first-class designed experience.
- Every new template produced for the platform must implement both blocks. A template that only implements ws_content breaks Mobile Mode for that feature.
- The Flutter mobile app is the long-term replacement for Mobile Mode in the browser. Mobile Mode remains as the web fallback when the Flutter app is unavailable.
- Both surfaces share the DESIGN.md design language - the visual signature is consistent across both.

**ADR-016**

**Local-First as the Ichebo Architectural Philosophy**

| **Field** | **Value**                            |
| --------- | ------------------------------------ |
| Number    | ADR-016                              |
| Status    | APPROVED                             |
| Date      | May 2026                             |
| Authors   | Chizola (domain); Claude (technical) |

## **Context**

The Ichebo platform was built Version 1 and 2 as a cloud-first web application hosted on a Hetzner VPS. During the May 2026 ecosystem architecture brainstorm, a fundamental philosophical shift was identified: the device should be treated as the primary computer, and the cloud as coordination infrastructure. This is the local-first philosophy.

This philosophy has profound implications for product design, data architecture, and user experience - particularly given Ichebo's primary context of Sub-Saharan Africa where connectivity is inconsistent and the device is often the most reliable computing resource available to a community.

## **Decision**

**DECISION**

_Local-first is the governing architectural philosophy for all new Ichebo products from Version 3 onward. The device is the primary computer. The cloud is coordination. A community should be able to operate Ichebo Desktop fully and permanently without internet access and lose nothing of operational value._

## **What Local-First Means**

| **Principle**         | **What it means in practice**                                                                                                                  |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| Device is primary     | All data is written to local storage first. The cloud receives a copy - it does not own the original.                                          |
| Cloud is coordination | The cloud connects communities to each other, gives network leadership visibility, and enforces KGS compliance. It does not run the community. |
| Offline is default    | Every feature must work without internet. Online features are enhancements, not dependencies.                                                  |
| Sync is deliberate    | Data moves from device to cloud on explicit sync - not on every write. The community controls when their data leaves the device.               |
| Data sovereignty      | The community owns their data on their device. The cloud sees it only when they choose to sync.                                                |

## **What Local-First Does NOT Change**

Local-first applies to new products (Ichebo Desktop, Ichebo Mobile as native Flutter app). It does not apply retroactively to the existing Django cloud platform, which remains cloud-first and continues to operate as built through Version 2.

- The Django cloud platform (Versions 1 and 2) is cloud-first and remains so.
- The DRF API endpoints are unchanged - they become the sync target, not the primary data store.
- The data contract (v10 and v11) continues to govern cloud-side schemas.
- Local-first is an additive philosophy - it adds Ichebo Desktop and the Sync Engine to the ecosystem without removing or breaking what was built.

## **Strategic Rationale**

The local-first philosophy serves three strategic goals:

- Reliability in low-connectivity contexts - Ichebo must work in rural South Africa, Zimbabwe, Zambia, and across Africa where mobile data is expensive and power is unreliable. A community's governance data must not be hostage to an internet connection.
- Data sovereignty - apostolic communities have a legitimate concern about their member data, governance documents, and formation records being held on a cloud server they do not control. Local-first puts the data on a device the community controls.
- Competitive differentiation - no faith-based governance platform in the market is local-first. This is genuinely new ground and a meaningful product advantage.

## **Consequences**

- All new products in the Ichebo ecosystem (Desktop, native Mobile) are designed local-first from the ground up.
- The Sync Engine (ADR-018) is the technical expression of this philosophy - it is mandatory infrastructure for all local-first products.
- UUID primary keys everywhere is a non-negotiable consequence of local-first - records created offline must have permanent identities before they reach the cloud.
- Soft deletes everywhere is a non-negotiable consequence - the sync engine cannot reconcile hard deletes.

**ADR-017**

**Ichebo Desktop as the Primary Product in the Ichebo Ecosystem**

## **Context**

The Ichebo ecosystem now consists of three client surfaces: the Django web platform (cloud-first, Version 2 complete), the Flutter mobile app (in development), and a new product identified in the May 2026 brainstorm: Ichebo Desktop. The brainstorm addressed a strategic question directly: is Ichebo Desktop a companion to the cloud, or is the cloud a service to the Desktop?

## **Decision**

**DECISION**

_Ichebo Desktop is the primary product in the Ichebo ecosystem. The cloud platform exists to coordinate, verify, and give network visibility to what is already happening locally on every Ichebo Desktop installation. A community can run Ichebo Desktop permanently without internet access and lose nothing of operational value._

## **Product Definition**

Ichebo Desktop is a standalone, installable, offline-capable community operating system. It serves the community steward - the local pastor, coordinator, or administrator responsible for a single Sceptre Community.

| **Attribute**         | **Value**                                                                        |
| --------------------- | -------------------------------------------------------------------------------- |
| Technology            | Flutter - one codebase for Windows, macOS, and Linux                             |
| Database              | SQLite (local, embedded, zero config)                                            |
| Network requirement   | None for operation. Internet required only for sync and initial onboarding.      |
| Primary user          | Community steward / local pastor (Level 3+)                                      |
| Scope                 | Single community - one Sceptre Community per installation                        |
| Relationship to cloud | Syncs to Ichebo Cloud when internet available. Cloud does not run the community. |

## **MVP Scope**

Ichebo Desktop MVP is ruthlessly minimal. Three functions only:

- **People - member registry, competence levels, shepherd assignments**
- **Activity - log attendance, service participation, activities**
- **Sync - sync status, last synced, pending changes, conflict review**

Explicitly out of Ichebo Desktop MVP:

- Learn LMS and formation pathway tracking
- Paraclete intelligence layer
- Video and media
- Full KGS five-level permission complexity
- Multi-community hierarchy management (cloud handles this)

## **First-Run Experience**

Three screens only:

- Organisation identity - organisation name, location, tier
- Administrator account - name, email, password
- Connection choice - start fresh OR connect to existing Ichebo Cloud account (sync down initial data)

## **The Cloud as Passport Office**

Before Ichebo Desktop activates, a community must be KGS-verified. The onboarding flow:

- Community applies via Ichebo Cloud
- KGS compliance verified by authorised leader - human in the loop, not a checkbox
- Tenant created in cloud - position in network hierarchy locked
- Desktop licence key generated and issued
- Community installs Ichebo Desktop, enters licence key
- Desktop activates with verified identity and syncs initial data
- Community operates fully offline from this point

## **Information Architecture**

Five sections only:

| **Section** | **Contents**                                                             |
| ----------- | ------------------------------------------------------------------------ |
| Home        | Today's snapshot - upcoming gathering, pending tasks, sync status        |
| People      | Member registry, levels, shepherd assignments                            |
| Activity    | Log attendance, service, participation                                   |
| Governance  | View local handbook and mandates (received from cloud, read-only in MVP) |
| Sync        | Status, last synced timestamp, pending changes count, conflict queue     |

## **Consequences**

- Ichebo Desktop is a new product in the Ichebo family - not a feature of the cloud platform.
- It requires the Sync Engine (ADR-018) and the Go foundation engines (ADR-019) as infrastructure.
- It is a Version 3+ deliverable - it does not conflict with or block Version 2 completion.
- The cloud platform (Django) is not deprecated by Ichebo Desktop - it serves a different role: network governance, apostolic oversight, KGS compliance.
- Dark mode is non-optional from day one - required for low-light church environments.

**ADR-018**

**Sync Engine as a Standalone Go Binary**

| **Field** | **Value**                            |
| --------- | ------------------------------------ |
| Number    | ADR-018                              |
| Status    | APPROVED                             |
| Date      | May 2026                             |
| Authors   | Chizola (domain); Claude (technical) |

## **Context**

The local-first philosophy (ADR-016) requires a mechanism for synchronising data between local device storage and the Ichebo Cloud. The data contract v10 contains a delta sync endpoint (GET /api/sync/changes/) designed for Flutter mobile. This is a different and simpler mechanism than what Ichebo Desktop requires.

Ichebo Desktop needs a full, bidirectional, conflict-aware sync engine that can: track every local change, push changes to the cloud in order, pull cloud changes down, resolve conflicts by data type rules, and run continuously in the background without blocking the UI. This cannot be adequately served by the existing delta sync endpoint alone.

## **Decision**

**DECISION**

_The Ichebo Sync Engine is a standalone Go binary - embedded in Ichebo Desktop and callable as a background service. It is the strategic secret sauce of the Ichebo ecosystem. It is owned entirely by Ichebo, not a third-party library. Over time it will be published as the Ichebo Sync Protocol - a standard for KGS-compliant community data synchronisation._

## **The Four Jobs**

| **Job** | **Description**                                                                                                                    |
| ------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| TRACK   | Know what changed, where, and when. Every write to local SQLite appends a row to the ChangeLog table. This is the engine's memory. |
| PUSH    | Send local changes to the cloud when internet is available. Idempotent, batched, ordered by changed_at ascending.                  |
| PULL    | Receive cloud changes and write them locally through the appropriate engine (Records, Activity, or Relationships).                 |
| RESOLVE | Decide what wins when both sides changed the same record. Rules-based by data type - no user intervention in most cases.           |

## **ChangeLog Table Schema**

| **Field**    | **Description**                                                         |
| ------------ | ----------------------------------------------------------------------- |
| id           | UUID - unique identifier for this change event                          |
| entity_type  | String - e.g. "member", "activity", "record"                            |
| entity_id    | UUID - the record that changed                                          |
| operation    | Enum - CREATE \| UPDATE \| DELETE                                       |
| changed_at   | Timestamp - when it happened locally                                    |
| synced_at    | Timestamp - null until successfully pushed to cloud                     |
| device_id    | UUID - which installation made this change                              |
| payload_hash | String - checksum of the data at time of change, for conflict detection |

## **Conflict Resolution Rules**

| **Data type**                                | **Rule**                     | **Reason**                                                                 |
| -------------------------------------------- | ---------------------------- | -------------------------------------------------------------------------- |
| Governance records - Handbook, Mandates      | Cloud wins always            | Authority flows from apostolic leadership downward. Local cannot override. |
| Permissions and roles                        | Cloud wins always            | Authority flows downward, never upward.                                    |
| Community settings                           | Cloud wins always            | Set at onboarding, controlled by network.                                  |
| Member registry - names, levels, assignments | Last write wins by timestamp | Steward updates are legitimate from both device and cloud.                 |
| Activity logs                                | Merge - both versions kept   | An activity logged offline and one logged online are both real events.     |
| Personal records - journal, Bible notes      | Local wins always            | Personal and private to the device.                                        |

## **Non-Negotiable Data Rules**

- **UUID primary keys everywhere - both local SQLite and cloud PostgreSQL. No auto-increment integers. Ever. A record created offline must have a permanent identity before it reaches the cloud.**
- **Soft deletes only - no hard deletes anywhere in the system. deleted_at timestamp is set. ChangeLog records a DELETE operation. The cloud decides whether to honour it.**
- **SQLite WAL mode - PRAGMA journal_mode=WAL. Non-negotiable for safe concurrent access between UI process and background sync process.**
- **Idempotency - every push operation must be safe to run twice without creating duplicates. Same UUID, same operation - cloud ignores the duplicate.**
- **Device identity - a stable UUID generated on first run, stored locally. This is how the cloud knows which node sent which change.**

## **Sync Triggers**

- On connectivity detected - the app monitors network state. When it transitions from offline to online, sync starts automatically in the background.
- On app open - if online, sync runs silently on launch.
- Manual - the user can tap "Sync now" from the status bar at any time.

Sync never blocks the UI. It runs in a background goroutine. The user keeps working.

## **Sync Status - Four States**

| **State** | **Display**                           |
| --------- | ------------------------------------- |
| Synced    | ● Synced - 2 minutes ago \[Sync now\] |
| Offline   | ○ Offline - 47 changes pending        |
| Syncing   | ⟳ Syncing - 23 of 47 changes...       |
| Conflict  | ⚠ 3 records need attention \[Review\] |

## **Relationship to Delta Sync Endpoint**

The existing GET /api/sync/changes/ endpoint in the data contract (Part 17) is retained for the Flutter mobile app. It is a simpler, read-only pull mechanism. The full Sync Engine (this ADR) is the bidirectional, conflict-aware engine for Ichebo Desktop. Both coexist. They use the same cloud API endpoints where possible but the Sync Engine requires two new dedicated endpoints: POST /api/sync/push/ and GET /api/sync/pull/?since=.

## **Build Sequence**

This sequence is non-negotiable. UI features are built after the data layer is solid.

- Design UUID schema - both SQLite and PostgreSQL models
- Build ChangeLog table and write trigger - every local write appends here
- Build soft delete pattern - no hard deletes anywhere
- Build Records Engine Go module (ADR-019)
- Build Activity Engine Go module (ADR-019)
- Build Relationships Engine Go module (ADR-019)
- Build POST /api/sync/push/ endpoint on cloud
- Build GET /api/sync/pull/?since= endpoint on cloud
- Build sync engine background process - push/pull/resolve using Go engine modules
- Build conflict resolution rules
- Build ConflictQueue and user review UI
- Build sync status bar - four states
- 7-day offline test - everything reconciles correctly after reconnection

## **Consequences**

- The Sync Engine is the moat. It must be built before any UI feature of Ichebo Desktop.
- The Sync Engine as a Go binary can be embedded in Ichebo Desktop, run as a cloud coordination service, and eventually published as an open protocol.
- The Sync Engine orchestrates the Records Engine, Activity Engine, and Relationships Engine during sync - it does not write raw data directly.

**Go as the Language for Ichebo Foundation Engines**

| **Field** | **Value**                            |
| --------- | ------------------------------------ |
| Number    | ADR-019                              |
| Status    | APPROVED                             |
| Date      | May 2026                             |
| Authors   | Chizola (domain); Claude (technical) |

## **Context**

The current Ichebo platform implements domain logic (records management, activity tracking, relationship traversal) as Django apps in Python. This coupling of domain logic to the web framework means the logic cannot be reused in Ichebo Desktop, cannot be independently tested, and cannot be independently deployed. The local-first philosophy (ADR-016) and the Ichebo Desktop product (ADR-017) require portable, embeddable engine implementations.

## **Decision**

**DECISION**

_Go is the language for all Ichebo foundation engine implementations. The engines are standalone Go modules - embeddable in Flutter Desktop via FFI, runnable as cloud microservices, independently testable. Python/Django remains the cloud web layer and API adapter. The engines are the domain logic layer. Django calls the engines - Django does not own the logic._

## **Why Go**

| **Property**   | **Go advantage**                                                                                         |
| -------------- | -------------------------------------------------------------------------------------------------------- |
| Concurrency    | Goroutines handle background sync while the user works - native, lightweight, no blocking.               |
| Deployment     | A Go binary is a single executable file. No runtime dependencies, no virtualenv, starts in milliseconds. |
| Cross-platform | Compiles to Windows, macOS, Linux, Android (via gomobile) from one codebase.                             |
| Embeddability  | Go binaries embed cleanly into Flutter via FFI bridge. The engine runs inside the desktop app.           |
| Independence   | Each engine is independently testable, deployable, and versionable.                                      |
| Performance    | Fast enough for all Ichebo use cases with significant headroom. Memory-efficient on modest hardware.     |

## **The Six Foundation Engines**

| **Engine**           | **Responsibility**                                                                                                                                                                          |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Records Engine       | Create, read, update, soft-delete records. Enforce record_class, record_family, record_type rules. Enforce the four mandatory fields. Works against SQLite locally and PostgreSQL on cloud. |
| Activity Engine      | Log participation events. Track ActivityLog entries. Calculate progress. Recurring activity logic. Completion signals. Portable - same rules, same data shapes, both environments.          |
| Relationships Engine | Create, read, traverse relationships. Direction-aware (directed vs undirected). HRS methodology. bible_verse_id links. derived_from chains. Version history chains.                         |
| Bible Engine         | Scripture text, translations (KJV, ASV, WEB), reference resolution. Returns text only - no HRS logic lives here.                                                                            |
| Calendar Engine      | Governance cycles, Annual Operational Rhythm, programme milestones, event coordination. Events are expressions of the KGS operational rhythm - not standalone records.                      |
| Sync Engine          | ChangeLog management, Push/Pull/Resolve. Orchestrates the other engines during sync. See ADR-018.                                                                                           |

## **The Three-Phase Separation Plan**

**Phase 1 - Clean the Boundaries (Current / Version 2 → 3 transition)**

Keep Django but stop adding domain logic to it. Extract all business rules into Python modules that Django calls - not Django doing the logic itself. Django becomes the thin HTTP layer that calls modules. This is the preparation phase - no Go yet.

**Phase 2 - Extract Engines (Ichebo Desktop Build)**

As Ichebo Desktop is built, the Go engines are built properly. The cloud Django app begins calling the same engine specifications via internal API or by Python-Go bridge. Django becomes a thin adapter over Go engines. The Sync Engine is built in Go and embedded in Desktop.

**Phase 3 - Service Extraction (As Products Mature)**

As each product's complexity demands it, engines are promoted to independently deployed services. The API Gateway is formalised. Clients do not know which service answers their request. Community social graph extracted when it needs to scale. Calendar Engine formalised as independent service.

## **Engine-to-App Mapping**

| **App / Product**                  | **Engines called**                                                          |
| ---------------------------------- | --------------------------------------------------------------------------- |
| Bible App / Ichebo Bible           | Bible Engine (text only)                                                    |
| Learn App / Ichebo Formation       | Records Engine + Activity Engine (progress)                                 |
| Activity App                       | Activity Engine + Records Engine                                            |
| Community App / Ichebo Community   | Records Engine + Activity Engine + Relationships Engine                     |
| Governance App / Ichebo Governance | Records Engine + Relationships Engine (version chains, HRS)                 |
| Ichebo Handbook                    | Records Engine + Relationships Engine + Bible Engine (HRS scripture module) |
| Paraclete                          | All engines - read-only, intelligence layer                                 |
| Sync Engine                        | Orchestrates Records + Activity + Relationships engines                     |

## **The Rust Question**

Rust would produce faster, more memory-efficient, more provably correct engines. Its ownership model makes entire classes of data corruption errors impossible at compile time. The recommendation: write the engines in Go first. Get them working, tested, and running in production. If a performance wall is hit - rewrite the hot path in Rust. Go now, Rust when the stars align.

## **Language-Agnostic Specifications**

Each engine must be defined as a language-agnostic specification - data shapes, rules, and behaviours that both Go (desktop/mobile) and Python/Django (cloud) implement independently. The data contract v10 (and v11 when produced) is the starting point for these specifications. Both implementations must produce identical results for identical inputs. The data contract is the lingua franca.

## **Consequences**

- Django is not deprecated. It is recharacterised: it is the web layer and API adapter, not the domain logic owner.
- No new domain logic is added to Django. Business rules go into Python modules (Phase 1) and eventually Go engines (Phase 2).
- The Records Engine in Go must enforce identical rules to the Django records app. The data contract governs both.
- Each Go engine is independently compilable, independently testable, and has its own version.
- The Flutter-to-Go bridge uses FFI for embedded engine calls within Ichebo Desktop.

**ADR-020**

**Ichebo Handbook as a Standalone Product - Supersedes Handbook-as-Tenant**

| **Field**  | **Value**                                                                                   |
| ---------- | ------------------------------------------------------------------------------------------- |
| Number     | ADR-020                                                                                     |
| Status     | APPROVED                                                                                    |
| Date       | May 2026                                                                                    |
| Authors    | Chizola (domain); Claude (technical)                                                        |
| Supersedes | Part 1 architecture statement in data contract v10 (Handbook as prime tier:handbook tenant) |

## **Context**

The data contract v10 contains a locked architectural decision: "Handbook as a prime tier:handbook tenant at /global/handbook/ with Level 5 write access only (Level 4+ read)." This decision was made as a pragmatic workaround to enable Handbook functionality within the existing tenant hierarchy model.

The Version 2 build of The Desk (ADR-014) and the May 2026 ecosystem architecture brainstorm produced a clear conclusion: the Handbook-as-tenant model is architecturally incorrect. The Handbook is not a tenant. A tenant is a community node in the network. The Handbook is the institutional memory and governing intelligence that precedes all tenants, authorises them, and governs them.

## **The Correct Understanding of the Handbook**

The KGS framework establishes the Handbook as:

- The mandate documents - why the system exists, the Kingdom purpose
- The HRS (Hierarchy and Relationship System) - the interpretive methodology for all governance
- The prophetic keys - scripture-governance mappings, keys library
- The governance principles - the seven-pillar structure, authority flows
- The version history of all governing documents

These are not records in a tenant. They are the foundation that gives rise to all tenants. The correct mental model:

| **Layer**           | **What it is**                                                                                    |
| ------------------- | ------------------------------------------------------------------------------------------------- |
| Handbook            | Institutional memory - mandate, HRS, prophetic keys, governance principles. Precedes the network. |
| KGS Framework       | The governing framework published from the Handbook to the network.                               |
| Network of Tenants  | Communities that receive and operate under the framework.                                         |
| Apostolic Authority | Persons who steward the Handbook and govern the network.                                          |

## **Decision**

**DECISION**

_Ichebo Handbook is a standalone product - not a tenant, not an app within another product, not a special-case tier in the tenant hierarchy. The Handbook-as-tenant implementation (tier:handbook at /global/handbook/) remains in production for Version 2 as a working workaround and will be migrated to the standalone Handbook product when Ichebo Handbook is built. This ADR formally supersedes the Handbook-as-tenant locked decision in data contract v10 Part 1._

## **What Ichebo Handbook Is**

- **A fully-fledged standalone product - Ichebo Handbook**
- **Its own service with its own data domain**
- **Access by invitation only - Level 5 architects and designated stewards**
- **Contains: mandate documents, governance principles, HRS library, prophetic keys, scripture-governance mappings, full version history of all governing documents**
- **The HRS methodology lives here as a first-class citizen - not as a relationship type in the records table**
- **Publishes downward to the network - communities receive Handbook outputs, they do not edit the Handbook**
- **The Desk (ADR-014) informs the Handbook's authorship interface - the editorial canvas pattern is proven and carries forward**

## **Bible Engine and HRS - The Correct Separation**

The Handbook contains an HRS Scripture module - the advanced tool for architects to map scripture to governance documents. This is distinct from the Bible App's personal reading functionality:

| **Surface**                  | **Relationship with scripture**                                                                                                                                                   |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ichebo Bible (product)       | Personal scripture engagement - reading, notes, highlights, reading plans. Calls Bible Engine for text. No HRS functionality.                                                     |
| Ichebo Handbook (HRS module) | Advanced scripture mapping for architects. Links passages to governance documents. Version-controlled, audited, Level 5 only. Calls Bible Engine for text, adds HRS layer on top. |

The Bible Engine provides text to both. What each product does with that text is completely different.

## **Migration Path for Handbook-as-Tenant**

The existing Handbook tenant at /global/handbook/ remains in production through Version 2. Migration to Ichebo Handbook as a standalone product will:

- Export all existing Handbook records from the tenant context
- Import into the new Handbook product's data domain
- Remove the handbook tier from the tenant model
- Update all references in templates and API endpoints
- Produce a data contract amendment when migration is planned

## **Consequences**

- The Handbook-as-tenant decision in data contract v10 Part 1 is superseded by this ADR. A v11 amendment note acknowledges this.
- No new features are built into the Handbook tenant. It is maintained as-is until migration.
- Ichebo Handbook as a standalone product is a Version 3+ deliverable.
- The design pattern of The Desk directly informs Handbook's authorship interface.
- The HRS scripture mapping tool lives exclusively in Ichebo Handbook - it is not built into the Bible App.

**ADR-021**

**Ichebo Media as a Standalone Product with Purpose-Built Video Engine**

| **Field** | **Value**                            |
| --------- | ------------------------------------ |
| Number    | ADR-021                              |
| Status    | APPROVED                             |
| Date      | May 2026                             |
| Authors   | Chizola (domain); Claude (technical) |

## **Context**

The master roadmap v6 plans video support as: URL video embed (YouTube, Vimeo, direct .mp4) in Version 2, and defers self-hosted video to "Version 3+ if needed - MinIO + ffmpeg + HLS." This characterisation significantly underestimates the scope and architectural significance of video in the KGS context.

Video in the KGS serves three distinct purposes: teaching video (sermon recordings, ministry content), learning video (Academy curriculum delivery), and live streaming (gathered meetings, network-wide broadcasts). Each has completely different technical requirements. None of these can be adequately served by Django. Attempting to do so produces fragile, inadequate results.

## **Decision**

**DECISION**

_Ichebo Media is a standalone product with its own purpose-built Video Engine. It is not a Django app, not a feature, and not a "if needed" future consideration. It is a first-class product in the Ichebo ecosystem that requires proper engineering. "Self-hosted video if needed" is retired - Ichebo Media is planned and will be built._

## **What Ichebo Media Is**

Ichebo Media serves three video pipelines:

| **Pipeline**   | **Description**                                                                                                                     |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| Teaching video | Sermon recordings, ministry content, apostolic broadcasts. Upload, transcode, store, serve. VOD library. Accessible to all members. |
| Learning video | Academy curriculum delivery. Chapter markers, completion tracking via Activity Engine, offline download for Ichebo Desktop.         |
| Live streaming | Gathered meetings, network-wide broadcasts. RTMP ingest, HLS delivery, DVR recording, archive on stream end.                        |

## **Video Engine Architecture**

| **Component**     | **Technology**                                                     |
| ----------------- | ------------------------------------------------------------------ |
| Upload pipeline   | Go service - chunked upload handler, queue manager                 |
| Transcoding       | FFmpeg - multiple quality outputs, thumbnail generation            |
| Object storage    | Hetzner Object Storage (S3-compatible) - ics-media bucket          |
| CDN delivery      | CDN layer over Hetzner storage - HLS segment delivery              |
| Live ingest       | MediaMTX or nginx-rtmp - RTMP ingest server                        |
| Live delivery     | HLS/DASH from ingest - adaptive bitrate to viewers                 |
| Playback client   | HLS player in Flutter - quality adaptive, offline download capable |
| Progress tracking | Activity Engine - lesson completion events from video progress     |

## **Relationship to Existing Video Approach**

The Version 2 Video/Live app (URL-based video embed only) remains in production and is valid for its scope. Ichebo Media does not replace it immediately - it supersedes it as the platform's video infrastructure matures. The Version 2 approach (YouTube/Vimeo iframe embed) is the pragmatic starting point. Ichebo Media is the destination.

## **Consequences**

- The description "self-hosted video if needed" is retired from all documents. Ichebo Media is planned.
- Video processing cannot live in Django. Django serves the management UI - the Video Engine is Go + FFmpeg.
- Hetzner Object Storage (S3-compatible) is the storage layer. The ics-media bucket naming is established.
- Offline video download for Ichebo Desktop is a first-class requirement - not a nice-to-have.
- Live streaming requires MediaMTX or nginx-rtmp as a dedicated ingest server - this is separate infrastructure from the Hetzner VPS Django runs on.
- Ichebo Media is a Version 3+ deliverable - it does not block Version 2 or Ichebo Desktop MVP.

**ADR-001 UPDATE**

**Flutter Extended to All Client Surfaces Including Desktop**

| **Field** | **Value**                                                               |
| --------- | ----------------------------------------------------------------------- |
| ADR       | ADR-001 Amendment                                                       |
| Status    | APPROVED                                                                |
| Date      | May 2026                                                                |
| Amends    | Original Flutter-for-mobile ADR from 2026-04-25-ichebo-adr-version-2.md |

## **Original Decision**

ADR-001 (original): Flutter for mobile - Android-first. iOS later. Consumes DRF API. No HTMX on mobile.

## **Amendment**

**AMENDMENT**

_Flutter is the framework for all Ichebo client surfaces - not just mobile. This includes Ichebo Desktop (Windows, macOS, Linux), Ichebo Mobile (Android, iOS), and any future client surfaces. One codebase, all platforms, native rendering per platform._

## **Rationale**

Flutter's cross-platform capabilities extend to desktop targets (Windows, macOS, Linux) in addition to mobile. Ichebo Desktop (ADR-017) is built in Flutter. This means one design system (DESIGN.md), one component library, and one team capability serves all client surfaces. The DRF API endpoints that Flutter mobile already calls are the same endpoints Ichebo Desktop sync calls. The architecture is consistent.

## **Scope by Surface**

| **Surface**             | **Flutter target**                                                   |
| ----------------------- | -------------------------------------------------------------------- |
| Ichebo Desktop          | Windows, macOS, Linux - local-first, SQLite, Go sync engine embedded |
| Ichebo Mobile (Android) | Android - primary mobile target, DRF API, FCM push                   |
| Ichebo Mobile (iOS)     | iOS - after Android is stable, requires Apple Developer account      |

## **What Does Not Change**

- Android-first for mobile remains. iOS after Android is stable in production.
- The DRF API is the backend for both mobile Flutter surfaces.
- The Django web platform (Apostolic Command Shell) is not Flutter - it remains Django + HTMX for web browsers.
- DESIGN.md governs all Flutter surfaces - tokens.dart and app_theme.dart are derived from DESIGN.md.
